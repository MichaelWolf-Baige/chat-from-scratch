# 小模型预训练数据最佳方案：全面调研报告

> 日期：2026-06-27 | 项目：chat-from-scratch (100M 中文模型)
> 搜索范围：2024-2025 顶会论文 + 开源项目实战
> 覆盖：Phi-3 / SmolLM / TinyLlama / DCLM / BabyLM / YuLan-Mini / Steel-LLM / MiniMind / PROX

---

## 一、核心原则：小模型和大模型对数据的需求完全不同

大模型（7B+）可以用海量噪声数据"大力出奇迹"，但小模型（<1B）参数少、容量小，
**每条数据都必须 earn its place**。MiniMind 文档里说得最直白："Feed precisely, not massively."

**我们的实证**：同样 13M tokens，好的蒸馏数据 PPL=5，差的 MiniMind 等量 PPL=11。

---

## 二、数据配比：对 ~100M 模型的最优方案

### 学术界共识（基于 BabyLM、DCLM、YuLan-Mini、Steel-LLM）

| 数据类型 | 推荐占比 | 为什么 |
|----------|----------|--------|
| **中文通用文本**（百科/书籍/问答） | 50-60% | 构建语言基础能力 |
| **代码** | 10-15% | 提升推理能力，即使只做中文任务 |
| **数学/逻辑推理** | 5-10% | 近年共识：推理数据对泛化影响巨大 |
| **对话/QA** | 10-15% | 提供自然的语言交互模式 |
| **英文通用文本** | 5-10% | 保持跨语言能力，别全砍 |

### 经典配比参考

- 面向中文场景：**中:英:Code ≈ 8:1:1** 或 7:2:1
- YuLan-Mini（人大 2.4B）：分阶段数据课程 + 退火阶段数学/代码数据 → 1.08T tokens 达同级别 SOTA
- Steel-LLM（1B）：80% 中文，CEVAL 41.9 分，CMMLU 36 分

### ⚠️ 我们当前问题

80MB 蒸馏数据领域覆盖偏科严重——AI/学术类强，常识/地理/数学弱。
**不是数据量问题，是 prompt 领域覆盖问题。**

## 三、合成数据的黄金比例：20-30%

### 这是最关键的研究发现

| 研究来源 | 结论 |
|----------|------|
| Kang et al. (EMNLP 2025) | 1/3 合成 + 2/3 自然文本混合，训练速度提升 5-10 倍 |
| Du et al. (2025) | 小模型 (410M-1.4B) 对合成数据更敏感，**超过 30% 性能退化** |
| TinyLlama CPT 实验 | 20% 合成数据最优，40% 开始退化 |
| HuggingFace Codelion | **纯合成数据 → 泛化能力灾难性崩塌（717 倍差距！）** |

### 核心洞察

全合成数据缺失了自然语言的**噪声、多样性和不完美**。合成数据虽然
clean，但太"干净"了——模型在合成数据上学到的是简化后的世界模型，
遇到真实文本就崩。这就是为什么我们蒸馏模型常识类生成差的原因：
不是数据不够，是数据类型单一。

### 对我们的建议

**不要 100% 用蒸馏数据。** 改为：

```
50-60%：蒸馏数据（AI/学术/代码/数学领域为主）
20-30%：天然中文数据（维基百科、百度百科、知乎问答、书籍）
10-20%：MiniMind 数据中抽样的常识/对话/生活类文本
```

保留蒸馏数据的专业优势 + 弥补常识短板 + 避免纯合成的泛化崩塌。

## 四、数据处理 Pipeline：DCLM 验证的最佳流程

Apple DCLM 项目（NeurIPS 2024）用 412M 模型做数据实验，
验证了以下 pipeline 效果最好：

```
原始数据 → 语言识别 → 启发式过滤 → 去重 → 模型质量打分 → 配比混合 → 训练
```

### 具体步骤

| 步骤 | 操作 | 工具 |
|------|------|------|
| 1. 语言过滤 | 去除非中文文本 | fastText 中文分类器 |
| 2. 启发式过滤 | 去短文本(<30字)、去HTML残留、去重复字符 | 正则/规则 |
| 3. 精确去重 | 文档级 hash 去重 | SHA256 |
| 4. 模糊去重 | MinHash 近似去重（Jaccard >0.8 只留一个） | datasketch/SimHash |
| 5. 质量打分 | 用 0.3B 小模型对每条文档打分，去低分 | PROX 或 fastText |
| 6. 配比混合 | 按比例采样混合各数据源 | Python 脚本 |

### 关键发现

- **模型过滤是最关键的一步**——DCLM 发现用分类器做质量打分远比启发式规则有效
- 模糊去重时**优先保留旧版本**而不是新版本，效果更好
- 数据策略可以**在小模型 (412M) 上验证然后迁移到大模型 (7B)**——这对我们非常适合
- PROX 框架：用 **0.3B 小模型**做数据精炼，文本清理效果匹配人类专家

## 五、数据课程：Easy→Hard 一致获胜

多个独立实验（BabyLM 2024/2025、TinyLlama CPT、YuLan-Mini）都验证了：

| 策略 | 效果 |
|------|------|
| 随机混合 | 基线 |
| 分领域训练（先百科、再代码…） | ❌ 更差——灾难性遗忘 |
| **PPL 从低到高 (Easy→Hard)** | ✅ 一致最优 |
| Static Mixing（静态混合） | ✅ 也很强（HuggingFace Codelion 发现比 curriculum 好 2.7x） |

### 实操方案

在数据预处理阶段，用 Teacher 模型的 PPL 给每条数据打分：

```python
for doc in all_data:
    # PPL 越低 = 文本越可预测 = 简单样本
    doc.ppl = compute_perplexity(teacher_model, doc.text)

all_data.sort(key=lambda d: d.ppl)  # 从简单到困难
```

BabyLM 冠军方案验证：即使只排序 10M words，easy→hard 也能带来 +1-2 个点的提升。

### 注意：Hard cutoff 是灾难

HuggingFace Codelion 实验发现：**从合成数据突然切换到多样化数据**，
perplexity 差了 6 倍。过渡要平滑，不要硬切。

## 六、训练轮次：2-4 epochs 安全

Muennighoff et al. (2023) 数据约束 scaling law + Lovelace et al. (2025) 过拟合补偿模型：

| Epochs | 效果 | 建议 |
|--------|------|------|
| 1-4 | 几乎等价于等量全新数据，无过拟合 | ✅ 安全区 |
| 4-16 | 收益递减，过拟合风险出现 | ⚠️ 强 weight decay (λ=1.0) 可缓解 |
| 16+ | 收益归零或 loss 反弹 | ❌ 不划算 |

### 关键发现
- Strong weight decay (λ=1.0) 比标准 λ=0.1 能减少 ~70% 的过拟合系数
- 大模型比小模型更快过拟合重复数据
- 我们的 2 epochs 非常保守，数据质量好后可尝试 3-4 epochs

## 七、PROX：用小模型精炼预训练数据（ICML 2025）

### 核心思路

将数据精炼视为**编程任务**——让一个 0.3B 小模型学会"清洗文档"，
对每条文档生成定制化的 Python 操作：

```
文档级：drop_doc() / keep_doc()
块级（chunk-level）：remove_lines() / normalize() / keep_chunk()
```

### 关键结果

| 指标 | 提升 |
|------|------|
| 10 项基准平均 | +2.5% |
| Math 领域 Mistral-7B | +7.6% |
| CodeLlama-7B 数学 | +20.3% |
| 训练效率 | 只需 10B 精炼 tokens 匹配 200B 的原始 corpus |

### 对我们项目的启示

我们不一定直接装 PROX，但核心思路——**用廉价模型做数据打分和过滤**——可以直接用：
用 Qwen2.5-0.5B 对所有蒸馏数据做 PPL 评分，去异常值，质量归一化。

## 八、防幻觉：在蒸馏阶段解决，别留到训练后

### 为什么必须在数据阶段

基于 SWITCH (NAACL 2025)、Smoothed KD (2025) 的研究：

- **硬标签训练（标准交叉熵）本身就是幻觉的来源之一**——强制确定性
- 如果蒸馏数据包含 Teacher 模型编造的事实，学生模型学到的是"如何像 Teacher 一样自信地胡说"
- SFT/RLHF 只能抑制症状，修不了根

### 三种实操方法

#### 8.1 事实密集型 Prompt 加知识约束

```python
# ❌ 高风险（Teacher 可能胡编）：
"写一段关于北京的介绍"

# ✅ 低风险（给事实，Teacher 只做语言组织）：
"""根据以下已知信息，写一段关于北京的中文介绍：
- 北京：中国首都，面积16410平方公里
- 常住人口：约2188万（2023年）
- 地标：故宫、天坛、长城、颐和园
- 历史：建城3000余年，曾用名燕京、大都、北平
"""
```

这就是 RAG 蒸馏：先检索事实 → 强制 Teacher 基于事实生成。

#### 8.2 数学题双遍验证

```python
# 1: Teacher 生成题目 + 答案
# 2: 同一 Teacher 再算一遍
# 3: 两次结果不一致 → 丢弃整条
```

#### 8.3 事实验证标签

每条数据的 metadata 标记：
```json
{
  "text": "...",
  "fact_density": "high",
  "verifiable": true,
  "domain": "geography"
}
```

训练时对高事实密度数据额外加权。

## 九、BabyLM 挑战的核心发现

BabyLM 探索在**极度受限数据下**（10M-100M words，类似儿童接触的语言量）训练模型。

### 关键结果（2024-2025）

| 发现 | 详情 |
|------|------|
| **多样性 > 质量（对小模型）** | 混合语料 > 纯净语料，对 GPT2-18M/44M 模型 |
| **复杂度 > 简单度（对大模型）** | 经典书籍对 705M+ 模型最优，TinyStories 对 18M 最优 |
| **32K 词表是最优尺寸** | 太小（30K）和太大（50K）都更差 |
| **TV 对话数据有奇效** | 模仿现代媒体语言环境，BLiMP +3.4 pts |
| **MADLAD（高质量网页）反而有害** | 在低资源设定下"高质量"定义不同 |
| **Curriculum masking 有效** | 按词性习得顺序逐步开放训练 mask |

### 对我们的启示

- 100M 参数正好处在 BabyLM 研究的"分水岭"——多样性和复杂度都很重要
- **不要追求"最纯净"的数据**，适量"脏数据"反而有助于泛化
- 对话/多轮 QA 数据价值被低估了

## 十、TinyStories 的教训

### 能做

- 28M 参数的 LLaMA 模型在合成故事上能流利生成
- 证明了数据质量 > 数据量

### 不能做

- 纯合成数据有天花板——多样性不足
- **TinyStories 单独使用在所有 BabyLM benchmark 上都垫底**
- 只在 GLUE（传统 NLP 任务）上有竞争力
- 结论：合成数据是**拼图的一部分**，不能当全部

## 十二、推荐完整路线图

### 第一阶段：数据基建（当前优先级）

| # | 步骤 | 内容 | 预计耗时 |
|---|------|------|---------|
| 1 | 扩 prompt | 14 领域 × 200+ 关键词，50K+ 不重复 prompt | 0.5h |
| 2 | 蒸馏生产 | 加事实约束 + 数学双验，7 卡并行 | ~8h |
| 3 | 收集天然数据 | 维基中文 dump + 百度百科 + 知乎 | ~2h 下载 |
| 4 | 数据清洗 | 去重 + 质量过滤 + PPL 排序 | ~1h |
| 5 | 配比混合 | 60% 蒸馏 + 25% 天然 + 15% MiniMind 常识 | ~0.5h |

**产出**：~2-3GB 高质量混合数据，覆盖 14 领域，有事实约束，有 easy→hard 课程

### 第二阶段：超参搜索（和蒸馏并行）

```
GPU 1: 独立跑超参搜索矩阵（9 组配置 × 2 epochs × 13M tokens）
GPU 2-7: 蒸馏数据生产
```

### 第三阶段：验证

用优化后配置 + 新数据训练，对比当前 baseline (PPL=5)。

### 不急着上 5GB 的原因

1. 学界共识：数据质量 > 数据量，尤其对小模型
2. BabyLM 用 10M words 就能训练像样模型，我们已有 23M tokens
3. 纯合成数据有泛化崩塌风险，必须先验证混合策略
4. 2-3GB 先试点，方向对了再放大
